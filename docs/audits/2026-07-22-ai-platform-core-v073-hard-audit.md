## P0 — критические продуктовые разрывы

### P0.1. Каталог остаётся демонстрационным

В production две меры поддержки. Архитектура способна проверять десятки и сотни мер, но пользователь получает выбор из двух записей.

**Gate:** не заявлять массовый подбор до появления минимум 30 полностью структурированных мер и 10 мер с подтверждёнными Tier A требованиями.

### P0.2. Evidence Foundation не завершён

156 evidence-записей существуют, но 0 получили статус verified. 7 правил мер существуют, но 0 подтверждены.

**Gate:** положительный match и опубликованная Decision Card остаются запрещёнными.

### P0.3. Пользовательский интерфейс показывал внутреннюю модель данных

До v0.73 интерфейс выводил:

- JSON-подобные значения фактов;
- технические сетевые ошибки;
- длинные локаторы внутри основной карточки;
- полную матрицу требований без сворачивания;
- равноправные блоки без визуального приоритета;
- посимвольный перенос текста на узком экране.

**Исправление v0.73:** Product Workspace UI с маршрутом `Профиль → Документы → Результат`, сворачиваемыми доказательствами и человекочитаемыми ошибками.

### P0.4. Технические ошибки источников не должны быть содержанием продукта

В production встречаются timeout, DNS и TLS ошибки. Они полезны оператору, но не пользователю.

**Исправление v0.73:** категория и понятное действие показываются сразу; сырой текст скрыт в `Технических сведениях`.

## P1 — инженерный долг

### P1.1. Монолитные клиентские компоненты

До обновления:

- `TelegramProjectWorkspace`: 354 строки и семь обязанностей;
- `GovernmentOpportunityReport`: 546 строк и почти весь продуктовый отчёт в одном компоненте.

v0.73 улучшает структуру отображения, но следующий этап должен разделить:

- Telegram authentication;
- Project repository;
- Document upload queue;
- Fact review;
- Analysis runner;
- Measure workspace;
- Evidence workspace;
- Application route.

### P1.2. Supabase Security Advisor

Остаются предупреждения:

- `gi_clean_display_text` и `gi_normalize_change_event` без фиксированного `search_path`;
- публичное выполнение SECURITY DEFINER RPC;
- множество RLS-таблиц без явных policies.

Часть таблиц закрыта deny-by-default намеренно. Публичные агрегирующие RPC могут быть нужны интерфейсу, но для каждого должен существовать отдельный контракт данных, лимит строк и тест отсутствия внутренних идентификаторов.

### P1.3. Supabase Performance Advisor

Остаются:

- внешние ключи без покрывающих индексов;
- auth RLS initplan на `gi_projects`, `gi_project_documents`, `gi_project_checks`;
- дублирующиеся индексы;
- большое количество пока не использованных индексов.

Удалять unused indexes автоматически запрещено: проект имеет низкий production-трафик, поэтому статистика использования ещё недостаточна.

### P1.4. Нет отдельного полноценного Telegram E2E

CI проверяет TypeScript и production build, но не доказывает полный путь:

`Telegram initData → проект → файл → разбор → факт → анализ → Truth Gate → сохранённый отчёт`.

Нужен Playwright-контур с серверно сформированной тестовой Telegram-сессией в изолированной среде, без production user data.
