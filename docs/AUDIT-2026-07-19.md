# AI Platform Core — Factory Audit

Date: 2026-07-19
Status: RED / FOUNDATION ONLY

## Executive verdict

The repository is a useful architectural seed, but it is not yet a platform core. It currently provides static TypeScript contracts, an in-memory registry, an echo-style orchestrator, and three read-only HTTP endpoints.

## What is already good

- Dedicated repository with clear ownership.
- Node.js 20+ and strict TypeScript configuration.
- Explicit product and capability vocabulary.
- Separation between contracts, registry, orchestration, and HTTP entry point.
- No unnecessary framework or infrastructure sprawl.

## Critical gaps

### P0 — blocks a real vertical slice

1. The orchestrator does not execute actions. It returns the request payload unchanged.
2. The HTTP server does not expose the orchestrator.
3. No runtime validation exists for requests or registry data.
4. No action-handler registry or capability execution contract exists.
5. No tests or CI verify compilation and behavior.
6. Errors are unstructured strings and cannot be handled reliably by clients.

### P1 — required before product integration

1. No request timeout, cancellation, trace context, or execution metadata.
2. No configuration validation.
3. No authentication boundary.
4. No persistence or provider interfaces for memory, RAG, analytics, or billing.
5. Product IDs and capabilities are hard-coded into union types, making extension expensive.
6. No versioned API contract.

### P2 — required before production

1. No structured logging or observability.
2. No rate limiting, CORS policy, security headers, or body-size limits.
3. No release workflow, changelog, dependency scanning, or secret scanning.
4. No health distinction between liveness and readiness.
5. No deployment contract for Vercel or another runtime.

## Architectural correction

The platform must not attempt to implement every product. It must provide stable interfaces and execution infrastructure:

PRODUCT REQUEST
→ VALIDATE
→ RESOLVE PRODUCT
→ RESOLVE ACTION HANDLER
→ CHECK REQUIRED CAPABILITIES
→ EXECUTE WITH CONTEXT
→ COLLECT TRACE
→ RETURN TYPED RESULT

## Foundation v0.2 acceptance gates

- `npm run check` passes in CI.
- Unit tests cover accepted and rejected orchestration paths.
- `POST /v1/execute` validates and executes a registered action.
- Every result includes duration and trace ID.
- Unknown product, unknown action, invalid input, and handler failure return stable error codes.
- Main remains protected by pull-request workflow.

## Kill criteria

Do not add databases, vector stores, billing providers, agent frameworks, or UI packages until the foundation vertical slice passes all gates.
